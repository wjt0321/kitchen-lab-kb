# 样品库前端改版方案 — 借鉴 agent-cowork 界面结构

> 日期：2026-06-15  
> 前提：保持现有技术栈（HTML + Tabler CSS + 原生 JS），不引入新框架  
> 目标：吸收 agent-cowork 的界面分层思想，把样品库从“页面跳转型”升级为“工作台型”  
> 当前基线：`D:\kitchen-lab-kb` — FastAPI + Tabler CSS + 原生 JS + PyWebView 桌面壳

---

## 1. 核心结论

agent-cowork 前端最值得借的不是 Rust/GPUI 技术栈，而是它把界面当成**固定职责区 + 动态工作区**来设计的思路。

样品库当前是 **hash 路由切页面** 模式（`#/products` → 整页换，`#/products/1/edit` → 再换一页），改成**固定壳 + 面板切换**后，体验会更像桌面应用，同时完全不需要换技术栈。

---

## 2. 当前状态 vs 目标状态

### 2.1 当前状态

```
┌─────────────────────────────────────────┐
│  sidebar  │  整页内容（每次 hash 切换）    │
│           │                              │
│  产品列表  │  表格 / 表单 / 详情            │
│  配方记录  │  （三选一，全页替换）           │
│  成功率    │                              │
│  导出备份  │                              │
└─────────────────────────────────────────┘
```

问题：
- 从列表点到编辑 → 整个页面换掉，上下文丢失
- 从产品列表切到配方 → 又是全页替换
- 没有“工作区”概念，每次操作都是孤立的

### 2.2 目标状态（借鉴 agent-cowork）

```
┌──────┬─────────────────────────────────────────┐
│      │  titlebar：当前工作区标题 + 快捷操作        │
│ side │─────────────────────────────────────────│
│ bar  │                                         │
│      │  workbench（动态区）                       │
│ 导航  │  ┌ 产品列表 ┐ ┌ 产品编辑 ┐ ┌ 配方详情 ┐    │
│      │  │ （tab）  │ │ （tab）  │ │ （tab）  │    │
│      │  └─────────┘ └─────────┘ └─────────┘    │
│      │                                         │
│      │─────────────────────────────────────────│
│      │  statusbar：选中 N 条 · 上次备份 06-15    │
└──────┴─────────────────────────────────────────┘
```

核心变化：
- **不再全页跳转**：导航切换只换工作区内容
- **工作区支持多 tab**：列表、编辑、详情可以同时打开
- **底部有状态栏**：始终可见关键状态信息
- **sidebar 保持固定**：用户始终知道自己在哪

## 3. 借鉴 agent-cowork 的具体映射

### 3.1 Sidebar → 导航栏（已有，需要增强）

**agent-cowork 的做法：**
- 256px 固定宽度
- 搜索框 + 新建入口 + Pinned 区 + HOME 区 + workspace 菜单
- 会话可固定、可搜索、可双击重命名

**样品库可以怎么改：**

现有 sidebar 已经有导航链接。建议增强为：

```
┌─────────────┐
│ 🔍 搜索产品… │  ← 全局搜索（跳产品/配方）
│             │
│ 📦 产品管理  │  ← 主入口，始终高亮
│   ├ 全部产品 │
│   └ 已归档   │
│             │
│ 🧪 配方记录  │
│   ├ 全部试验 │
│   └ 成功率   │
│             │
│ ⚙️ 工具      │
│   ├ Excel导出│
│   ├ JSON导出 │
│   └ 一键备份 │
│             │
│ ─────────── │
│ 👤 用户名    │  ← 用户区（固定底部）
│ 🚪 退出     │
└─────────────┘
```

改动量：约 50-80 行 JS，不改 CSS 框架。

### 3.2 Workbench → 中央工作区（新概念）

**agent-cowork 的做法：**
- 34px tab bar，active tab 有 accent 条
- 内建页面：Plan / Todo / Diff / Swarm / README
- 文件 tab 与内建页面共存
- 底部 260px bottom panel 可切换

**样品库可以怎么改：**

把现在的 `#app` 容器升级为 tab 工作区：

```
┌──────────────────────────────────────────────┐
│ [📦 产品列表] [✏️ 编辑: 兴达-001] [📊 成功率]  │ ← tab bar
├──────────────────────────────────────────────┤
│                                              │
│  当前 tab 的内容                               │
│                                              │
│  ├ 如果是产品列表 → 表格 + 筛选 + 操作按钮      │
│  ├ 如果是编辑表单 → 表单组件                    │
│  └ 如果是详情页   → 详情卡片 + 配方列表          │
│                                              │
├──────────────────────────────────────────────┤
│ 📊 选中 12 条 · 上次备份 06-15 20:00 · 126 产品 │ ← statusbar
└──────────────────────────────────────────────┘
```

核心 JS 逻辑：

```javascript
// 工作区 tab 管理
const workbench = {
  tabs: [],           // [{ id, title, type, data }]
  activeTabId: null,

  openTab(tab) {
    // 如果已存在同 id 的 tab，切换到它；否则新建
  },

  closeTab(id) {
    // 关闭 tab，自动切到相邻 tab
  },

  renderTabBar() {
    // 渲染 tab bar，active tab 有高亮 accent
  },

  renderActiveTab() {
    // 根据 activeTab 的 type 渲染对应内容
  }
};
```

改动量估算：
- 新增 tab 管理模块：约 150-200 行
- 改造现有页面渲染函数：每个页面函数加一层 tab 包装
- 不改 API，不改后端

### 3.3 Composer → 输入区（可延后）

**agent-cowork 的做法：**
- 输入区不是一个单纯 textarea
- 有模式切换（Agent / Plan / Debug / Multitask / Ask）
- 有 model dropdown
- 有 TodoStrip

**样品库的应用场景：**

样品库没有“AI 输入”需求，但可以借鉴模式切换思路，用在**表单操作区**：

```
┌──────────────────────────────────────────────┐
│ [新增产品] [批量导入] [批量归档] [导出选中]     │ ← 操作模式切换
├──────────────────────────────────────────────┤
│  当前模式对应的操作面板                         │
└──────────────────────────────────────────────┘
```

这个可以放在列表页顶部，替代现在分散在各处的按钮。

改动量：约 80-120 行，在列表页顶部加一个操作栏组件。

### 3.4 Statusbar → 状态栏（新概念）

**agent-cowork 的做法：**
- 有独立的 `statusbar.rs` 文件
- 显示当前上下文状态

**样品库可以怎么用：**

在页面底部放一条固定状态栏：

```html
<div class="statusbar">
  <span>📊 选中 <strong id="sb-selected">0</strong> 条</span>
  <span>💾 上次备份 <strong id="sb-backup">--</strong></span>
  <span>📦 产品总数 <strong id="sb-total">--</strong></span>
  <span>🕐 <span id="sb-time">--</span></span>
</div>
```

改动量：约 30 行 HTML + 40 行 JS。

### 3.5 Timeline 折叠 → 配方试验历史

**agent-cowork 的做法：**
- 最终答案永远完整显示
- 中间步骤折叠
- 同类步骤聚合
- 展开/收起 toggle

**样品库的应用场景：**

产品详情页目前显示配方列表。可以参考这个思路优化配方试验历史的展示：

```
产品详情：兴达-001
──────────────────────────────────
配方试验历史（共 8 次）：

▼ 2026-06-10 ~ 06-15  （3 次试验，2 成功）
  ├ ✅ 06-15 配方A-改  原料: A(10g) B(5g) C(2g)
  ├ ❌ 06-12 配方A-原  原料: A(10g) B(3g)
  └ ✅ 06-10 配方A-初  原料: A(8g) B(5g)

▶ 2026-06-01 ~ 06-09  （5 次试验，3 成功）  ← 折叠

──────────────────────────────────
```

改动量：约 100-150 行 JS，纯前端展示逻辑。

## 4. 不改什么

明确不动的部分，避免过度改造：

| 不改 | 原因 |
|---|---|
| 后端 API | 不需要动，前端改版不碰接口 |
| 数据库 | 数据模型不变 |
| Tabler CSS | 继续用，只加少量自定义 |
| PyWebView 壳 | 桌面窗行为不变 |
| 现有路由逻辑 | 内部保留兼容，外部 hash 路由继续可用 |
| 备份/导出 | 功能不变，只挪位置 |

## 5. 实施顺序建议

按收益/成本排序，建议分三阶段：

### 阶段一：最赚的三件事（改动小、体验提升大）

1. **底部状态栏** — 30 行 HTML + 40 行 JS，半小时能做完
2. **操作栏模式切换** — 替代分散按钮，80-120 行 JS
3. **配方试验折叠视图** — 产品详情页最直观的改进

### 阶段二：核心结构改造

4. **工作区 tab 化** — 这是最大的一块，约 200-300 行 JS
5. **sidebar 增强** — 全局搜索 + 子菜单 + 用户区固定

### 阶段三：锦上添花

6. **快捷键** — `/` 聚焦搜索、`Ctrl+N` 新增产品
7. **批量操作面板** — 利用操作栏模式切换
8. **键盘导航** — Tab 区 Ctrl+W 关闭、Ctrl+Tab 切换

## 6. 代码组织建议

当前 `app.js` 1561 行，所有逻辑在一个对象里。建议在改造时分几个文件：

```
static/
├── app.js           # 入口：init / 路由 / shell 管理
├── workbench.js     # tab 管理 / 面板切换
├── pages/
│   ├── products.js  # 产品列表 / 表单 / 详情
│   ├── recipes.js   # 配方列表 / 表单 / 详情
│   └── tools.js     # 导出 / 备份
├── components/
│   ├── sidebar.js   # 导航渲染
│   ├── statusbar.js # 状态栏渲染
│   └── modal.js     # 弹窗封装
└── style.css        # 保持现有
```

不强制一次性拆完，改到哪个模块就拆哪个。

## 7. 和 agent-cowork 的差异对照表

| 维度 | agent-cowork | 样品库改版后 |

## 8. 风险与边界

### 不做的事

- 不引入任何 npm 包或前端构建工具
- 不引入 React / Vue / Svelte
- 不改成 Rust 原生桌面
- 不改后端 API
- 不增加 pip 依赖

### 要注意的

- tab 切换时注意内存：长时间打开多个编辑 tab 可能积压未保存状态，建议加“未保存提醒”
- 全局搜索需要后端支持一个 `/api/search?q=xxx` 接口（或前端直接遍历已有数据）
- PyWebView 窗口尺寸较小（约 1024×768），多 tab 时不要做太宽的布局

---

## 9. 总结

> **不换技术栈，只换界面组织方式。**

agent-cowork 给样品库的最大启示不是 Rust，也不是 GPUI，而是：

- 把界面拆成**固定职责区**
- 让用户在一个窗口里**不跳页完成多步操作**
- 把状态、导航、操作、内容各自找到**稳定位置**

这些完全可以用你现在这套 HTML + Tabler CSS + 原生 JS 实现，不需要推翻重来。

# Kitchen Lab Layout Hardening Design

## 背景

当前 `kitchen-lab-kb` 的工作台已经具备完整业务能力，但最近一轮界面升级后暴露出一组明显的桌面端体验问题：

- 左侧栏在窄屏时会被挤到页面上方，破坏工作台结构。
- 外链图标字体资源被浏览器拦截，控制台持续出现 `ERR_BLOCKED_BY_ORB`。
- 主内容区宽度受限、留白偏多，列表与卡片没有充分利用横向空间。
- 分页按钮过小，选中态和当前态的视觉反馈不明显。
- 侧栏、顶部区、状态栏和主内容区的职责边界不够清晰，滚动时缺少“固定工作台”的稳定感。
- 左侧栏工具入口与顶部工具入口重复，降低信息架构清晰度。

本次设计聚焦以下前端文件：

- `templates/base.html`
- `static/style.css`
- `static/app.js`

本次改造不涉及后端 API、数据库结构、导入导出逻辑和核心路由语义。

## 目标

### 主要目标

- 将当前页面收敛成稳定的桌面工作台结构。
- 解决窄屏下左侧栏挤压到顶部的问题。
- 取消对外链图标字体的依赖，消除 `ORB` 报错。
- 提升主内容区的信息密度，让列表、筛选和操作区更实用。
- 强化当前项高亮、圆角和可点击反馈。
- 清理重复功能入口，明确“左侧导航、顶部操作、底部状态”的分工。

### 非目标

- 不新增业务模块。
- 不重写为新前端框架。
- 不改动后端接口返回结构。
- 不重新设计产品、配方、成功率等业务流程。
- 不追求移动端独立体验体系，只保证窄屏可用且结构合理。

## 用户确认的关键决策

### 1. 窄屏侧栏行为

用户确认：窄屏时不再让 `aside` 挤到页面上方，而是改为抽屉菜单。

这意味着：

- 桌面端保持左侧固定导航。
- 窄屏时通过左上角菜单按钮展开侧栏。
- 主内容区在窄屏下仍保持主阅读优先。

### 2. 工具入口保留位置

用户确认：保留顶部工具区，删除侧栏工具分组。

这意味着：

- 侧栏只承担导航职责。
- 备份、导入、导出、退出统一保留在顶部操作区。
- 页面信息架构从“双处可操作”收敛为“单处系统操作”。

### 3. 滚动方式

用户确认：使用“主内容整体滚动”的方式。

这意味着：

- 左侧栏固定。
- 顶部工具栏固定。
- 底部状态栏固定。
- 中间主内容区独立滚动。

### 4. 改造方案

用户确认采用“方案 1 平衡改造”。

这意味着：

- 解决结构、图标、滚动、留白、分页、高亮和入口重复等问题。
- 保留现有工作台语言，不走极简到过度克制的方向。
- 优先以最小业务风险解决最大范围的视觉与交互问题。

## 设计原则

### 1. 导航、操作、状态分层

页面必须明确分成三层：

- 左侧导航层
- 顶部系统操作层
- 中间业务工作层

底部状态栏作为全局辅助层存在，但不与主业务层抢注意力。

### 2. 结构稳定优先于局部修饰

优先解决工作台外壳稳定性、滚动关系和信息架构，再处理圆角、按钮和高亮等局部体验。

### 3. 桌面效率优先

优化以桌面端为主，目标是让产品列表、筛选、分页、批量操作和详情跳转更顺手，而不是单纯追求“更好看”。

### 4. 去重优先

重复入口比缺少装饰更伤害可用性。凡是功能已经在顶部有完整入口，侧栏不再重复承载。

### 5. 密度提升但不拥挤

减少无效留白、放宽内容宽度、压缩头部装饰块，但保留清晰层次和操作分组。

## 信息架构

### 全局布局

工作台统一采用以下结构：

1. 左侧固定导航栏 `aside`
2. 顶部固定系统操作栏 `topbar`
3. 中间可滚动主内容区 `workbench-body`
4. 底部固定状态栏 `statusbar`

布局目标：

- 视觉上侧栏与主内容紧密衔接，不出现“两个区域之间断开”的空白感。
- 主内容区横向尽可能展开，优先服务表格和表单阅读。

### 侧栏职责

侧栏只保留：

- 品牌区
- 搜索框
- 主导航
- 子导航
- 登录用户摘要

侧栏删除：

- Excel 导出
- JSON 导出
- 一键备份

### 顶部栏职责

顶部栏统一保留：

- 当前用户
- 备份
- 导入
- 导出菜单
- 退出
- 窄屏菜单按钮

顶部栏成为唯一系统操作入口。

### 状态栏职责

底部状态栏继续保留，但语义更明确：

- 选中数量
- 上次备份时间
- 产品总数
- 当前时间

状态栏固定在底部，避免随列表滚动而消失。

## 视觉与组件设计

### 品牌区

当前 `span.brand-mark` 使用文字 `KL`。本次改为：

- 优先展示本地品牌图标或 `ico` 资源。
- 无法展示图标时再回退到简写文字。
- 品牌区尺寸、圆角和对齐方式更稳定。

目标是让左侧栏头部从“占位字母块”升级为真正品牌入口。

### 图标策略

当前项目通过外链 `Tabler Icons` webfont 资源渲染图标，存在 `ORB` 风险。

本次设计要求：

- 去掉外链图标字体 CSS。
- 改为本地可控的图标方案。
- 新方案必须覆盖现有高频图标位，不影响按钮、导航和状态展示。

候选形式可以是本地 SVG 精灵、本地内联 SVG，或统一的前端图标渲染辅助方法。

约束：

- 不能继续依赖 `jsdelivr` 图标字体。
- 不能引入复杂构建步骤。

### 侧栏导航高亮

当前选中态不够明显。本次设计要求：

- 一级导航和二级导航都使用更清晰的圆角底色。
- 当前项增加更强的背景、边框或阴影层级。
- hover 与 active 要明显区分。
- 搜索框、导航链接和分组标题保持统一圆角语言。

目标是让用户一眼看出“当前在什么页面、可点击项在哪里”。

### 顶部工具区

顶部工具区保持完整，但做更清晰的分组：

- 左侧：窄屏菜单按钮
- 右侧：用户、备份、导入、导出、退出

要求：

- 导出仍保留菜单形式。
- 退出保持危险操作视觉。
- 常用按钮尺寸统一，点击区域充足。

### 登录页位置

当前登录卡片在首屏中的垂直位置偏低。

本次设计要求：

- 登录框整体上移，优先落在首屏偏上中部。
- 保持标题、输入框和主按钮的聚焦关系不变。
- 上移后仍保留足够呼吸感，不能贴近顶部。

目标是让登录页在打开时更像“入口工作台”，而不是页面中部偏下的普通表单。

### Page Hero 与操作区

当前 `page-hero` 占用垂直空间偏大，影响主列表可视面积。

本次设计要求：

- 压缩 `page-hero` 高度和内边距。
- 保留标题、说明和筛选动作，但避免“展示型 Hero”过强。
- 让产品列表页尽快进入“搜索 + 操作 + 表格”的工作状态。

### 表格与卡片

表格区域应优先获取空间：

- 放宽主容器最大宽度限制。
- 减少内容区左右留白。
- 表格卡片尽量与侧栏视觉连贯。
- 卡片头、分页和操作条的宽度与内容区对齐。

## 交互设计

### 窄屏抽屉导航

当视口小于桌面阈值时：

- 左侧栏默认收起。
- 顶部出现菜单按钮。
- 点击按钮后侧栏以抽屉形式滑出。
- 抽屉打开时显示遮罩层。
- 点击遮罩或导航项后自动关闭。

目标是解决侧栏挤到页面顶部的问题，同时保留导航完整性。

### 固定区与滚动区

滚动模型统一为：

- `aside` 固定不动。
- `topbar` 固定不动。
- `statusbar` 固定不动。
- `workbench-body` 单独纵向滚动。

这样可以确保在长列表、长表单、成功率明细等场景下，用户始终能看到核心导航和系统操作。

### 分页交互

当前分页按钮只有很小的箭头，存在识别和点击问题。

本次设计要求：

- 上一页和下一页按钮显著增大。
- 增加文字、边框和间距，不能只依赖小图标。
- 页码状态需更易读。

分页栏整体要更像操作工具条，而不是零散的几个小控件。

### 当前态反馈

以下元素需要统一增强“当前态”：

- 左侧主导航
- 左侧子导航
- 搜索输入框聚焦态
- 状态筛选控件
- 分页按钮禁用态与可用态

### 顶部与侧栏功能去重

系统操作只保留在顶部区后，页面内不再出现第二套“导出 / 备份”结构。

这样可以减少：

- 用户判断成本
- 重复视觉噪音
- 后续维护时的双处修改风险

## 实现范围

### 允许改动

- 调整 `templates/base.html` 的外链依赖与工作台壳结构。
- 重构 `static/style.css` 中的布局、响应式、滚动、导航和分页样式。
- 重构 `static/app.js` 中的侧栏、顶部区、图标渲染和分页结构。

### 不允许改动

- 后端接口
- 数据库结构
- Hash 路由语义
- 导入导出业务逻辑
- 备份与退出接口逻辑

## 验证标准

### 功能验证

以下流程必须保持正常：

- 登录
- 产品搜索
- 产品分页
- 新增产品
- 批量归档
- 导入
- 导出
- 备份
- 退出
- 成功率页面跳转

### 体验验证

- 窄屏时侧栏不再挤到顶部。
- 主内容区横向空间明显增加。
- 左侧导航和选中态更容易识别。
- 分页按钮更容易看见和点击。
- 侧栏与主内容的关系更像统一工作台，而不是两个分离面板。

### 控制台验证

- 页面加载后不再出现 `https://cdn.jsdelivr.net/...tabler-icons.min.css` 的 `ERR_BLOCKED_BY_ORB` 报错。

## 风险与缓解

### 风险 1：图标替换影响面广

当前大量按钮和导航都依赖 `.ti` 类名。若直接替换不完整，容易出现大量空白图标位。

缓解方式：

- 先建立统一图标渲染方案。
- 以高频图标优先覆盖。
- 在无法立即替换的场景保留文本可读性，避免功能不可识别。

### 风险 2：固定布局与滚动冲突

若 `topbar`、`statusbar` 和 `workbench-body` 的高度关系处理不当，可能导致内容溢出或滚动异常。

缓解方式：

- 统一在工作台壳层解决高度分配。
- 桌面端和窄屏分别验证滚动行为。

### 风险 3：信息密度提升后出现拥挤感

若只压缩留白而不重组层级，页面会从“太松”变成“太挤”。

缓解方式：

- 先收紧结构性留白。
- 保留标题区、操作区和表格区的节奏分层。

## 结论

本次改造目标不是推翻现有工作台，而是在保持业务逻辑不变的前提下，完成一次“布局稳定化 + 交互收口 + 视觉密度优化”：

- 桌面端保持稳定固定工作台结构。
- 窄屏切换为抽屉导航。
- 顶部成为唯一系统操作入口。
- 主内容区获得更多可用空间。
- 图标依赖改为本地可控方案，消除外链报错。

完成后，`kitchen-lab-kb` 会从“能工作的单页管理界面”进一步收敛为“结构稳定、操作清晰、信息密度合理的桌面工作台”。
